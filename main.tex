% ── Abstract ──────────────────────────────────────────────────────────
\begin{abstract}
\noindent
AI agents execute real-world actions through tool-use frameworks, but no formal mechanism ensures they are \emph{authorized} to act---only that they are capable. We introduce \emph{authority models}, a runtime primitive that evaluates proposed agent actions against machine-checkable policies before execution, extending attribute-based access control with three-valued decisions, session-aware state, and delegation-scoped authorization for the agentic setting. We formalize the $\May$ judgment---a decision function over seven parameters returning $\Allow$, $\Deny$, or $\Escalate$---and prove three properties: that capability and authority are informationally independent (\emph{Can/May Separation}), that policy composition preserves denials (\emph{Deny-Monotonicity}), and that escalation to human authority cannot be silently overridden (\emph{Escalation Monotonicity}). We show that authority models achieve Enforcement Determinism $\mathrm{ED} = 1.0$ and Degradation Under Adversarial Pressure $\mathrm{DAP} = 0.0$ by construction, as policy evaluation is deterministic and external to the model. We present AuthorityBench, an evaluation framework measuring authorization compliance across 30 scenarios at three adversarial pressure levels, and describe a reference implementation.
\end{abstract}

\vspace{6pt}

\noindent\textbf{Keywords:} authority models, AI agents, runtime authorization, action boundary, deontic logic, tool-use safety

\section{Introduction}
\label{sec:intro}

An AI agent is instructed to ``clean up the development environment.'' It has tool access to delete files, drop databases, and terminate processes. The tool-use framework grants capability: every one of these actions is \emph{feasible}. But should the agent be \emph{allowed} to drop the production database? Who decides? Based on what policy? Evaluated when?

These questions have clear answers in traditional software---access control lists, role-based permissions, capability tokens---developed over fifty years of systems security research~\citep{lampson1971protection, harrison1976protection, sandhu1996rbac}. But AI agents have not inherited this infrastructure. Large language models now browse the web, execute shell commands, access databases, send emails, and call external APIs through standardized protocols~\citep{anthropic2024mcp, mcpspec2025}. Each tool call is a point where latent model reasoning becomes concrete, world-affecting action. At this \emph{action boundary}, no authorization check occurs. The gap is not theoretical: OpenAgentSafety finds that agents ``disregard authorization entirely''~\citep{vijayvargiya2025openagentsafety}. Agent-SafetyBench reports an average safety score of 38.5\% across 16 LLM agents~\citep{zhang2024agentsafetybench}. No existing benchmark measures authorization boundaries.

The structural reason is that current capability frameworks handle \Can{} but not \May{}. Kambhampati et al.\ established that LLMs cannot reliably self-verify their plans and require external verifiers for soundness~\citep{kambhampati2024llmmodulo, kambhampati2024tasksuccess}. We extend this insight: just as self-verification is not verification, \emph{self-authorization is not authorization}. An LLM that ``decides'' it is permitted to act provides no more guarantee of legitimate action than an LLM that ``decides'' its plan is correct. Authorization, like verification, requires an external check.

The capability-authority distinction is well-established in the security literature. Dennis and Van Horn distinguished capabilities from access rights in 1966~\citep{dennis1966programming}. Miller formalized the principle that possessing a capability token does not imply possessing the authority to use it~\citep{miller2006robust}. We argue that the AI agent community needs to import and extend this distinction for the agentic setting, where actions are open-ended, tool-mediated, and selected by probabilistic reasoning.

\vspace{4pt}
\noindent\textbf{Contributions.} This paper makes four contributions:
\begin{enumerate}[leftmargin=2em, itemsep=2pt]
  \item We formalize \emph{authority models} as a runtime authorization primitive for AI agents, defining the $\May$ judgment with seven parameters and three-valued output (\Cref{sec:authority}).
  \item We prove three formal properties---Can/May Separation, Deny-Monotonicity, and Escalation Monotonicity---and establish decidability for a bounded policy language (\Cref{sec:properties}).
  \item We introduce two metrics---Enforcement Determinism (ED) and Degradation Under Adversarial Pressure (DAP)---and present AuthorityBench, an evaluation framework with 30 authorization scenarios (\Cref{sec:evaluation}).
  \item We describe a reference implementation and position authority models within the access control, deontic logic, and agent governance literatures (\Cref{sec:implementation,sec:related}).
\end{enumerate}

\begin{samepage}
\noindent The central claim of the paper:

\begin{center}
\textit{Capability and authority are informationally independent. Knowledge of whether an agent \Can{} perform an action provides no information about whether it \May{}. Any system conflating these layers admits privilege escalation by capability acquisition.}
\end{center}
\end{samepage}

\section{Background and Problem Statement}
\label{sec:background}

\subsection{The Agent Action Pipeline}

Tool-using agents follow a common pattern: the LLM receives a prompt, reasons about the task, generates a structured tool call (function name + arguments), the framework dispatches the call, the tool executes, and the result is returned to the LLM. The Model Context Protocol (MCP) standardizes this interface~\citep{anthropic2024mcp, mcpspec2025}. At no point in this pipeline is authorization checked. The framework verifies that the tool \emph{exists} and the arguments are \emph{well-typed}, but not that the agent is \emph{permitted} to invoke it in this context.

\subsection{Current Approaches and Their Limitations}

Four families of approaches constrain agent behavior, but none addresses authorization as a formal property.

\begin{table}[!ht]
\centering
\caption{Current approaches to constraining agent behavior.}
\label{tab:approaches}
\small
\begin{tabularx}{\textwidth}{@{}l l X@{}}
\toprule
\textbf{Approach} & \textbf{Mechanism} & \textbf{Why it is insufficient for authorization} \\
\midrule
Prompt constraints & System prompt rules & Non-binding. Stochastic compliance. Cannot be audited or version-controlled. Defense prompts alone are insufficient~\citep{zhang2024agentsafetybench}. \\
Alignment / RLHF & Training-time preferences & Shapes behavior, not delegation. ``Wanting to help'' $\neq$ ``authorized to act.'' Cannot express per-principal, per-context constraints~\citep{pierucci2026institutionalai}. \\
Output monitoring & Post-hoc detection & Too late for irreversible actions. Catches violations retroactively, not preventively~\citep{wang2025agentspec, chen2025shieldagent}. \\
Platform guardrails & Provider-specific filters & Narrow scope. Provider-locked. Constrain content, not authorization. A guardrail blocks toxic text; it does not block an unauthorized purchase order. \\
\bottomrule
\end{tabularx}
\end{table}

\subsection{Desiderata}

A solution to the authorization problem must satisfy six requirements:
\begin{enumerate}[leftmargin=2em, itemsep=1pt]
  \item \textbf{Deterministic.} Same request, same policy, same decision. No stochastic variance.
  \item \textbf{Pre-execution.} Evaluated before the tool call executes, not after.
  \item \textbf{External.} Not embedded in the LLM's context. The agent cannot override the check.
  \item \textbf{Composable.} Multiple policies combine with predictable semantics.
  \item \textbf{Delegable.} The principal who authorizes and the agent who acts are distinct entities.
  \item \textbf{Escalable.} ``Uncertain'' is a valid output---defer to human judgment rather than default to execution.
\end{enumerate}

\subsection{Threat Model}

We consider an agent with tool access acting on behalf of a principal, where the agent may: (a)~exceed its delegated scope, (b)~violate budget, temporal, or tenant constraints, (c)~be manipulated via prompt injection into unauthorized actions, or (d)~fail to escalate when policy is ambiguous. We assume honest principals---the threat is unauthorized \emph{agent} action, not principal-agent collusion.

\section{Authority Models}
\label{sec:authority}

\noindent\textbf{Notation.} Let $\mathcal{P}$ be the set of principals, $\mathcal{G}$ the set of agents, $\mathcal{A}$ the set of actions, $\mathcal{R}$ the set of resources, and $\Pi$ a finite set of policies. We write $\mathcal{D} = \{\Allow, \Deny, \Escalate\}$ for the decision domain.

\subsection{The \texorpdfstring{$\May$}{May} Judgment}

\begin{definition}[Authority Model]
\label{def:authority-model}
An \textbf{authority model} is a machine-checkable representation of the conditions under which an agent's action is \emph{properly constituted}---attributable to a principal, within delegated scope, satisfying contextual constraints, and subject to review, revocation, and audit.
\end{definition}

The core evaluation function is the $\May$ judgment:

\begin{equation}
\May(P, G, \alpha, R, C, H, \Pi) \;\to\; \mathcal{D}
\label{eq:may}
\end{equation}

\noindent where the seven parameters are:

\begin{table}[!ht]
\centering
\caption{Parameters of the $\May$ judgment.}
\label{tab:may-params}
\small
\begin{tabularx}{\textwidth}{@{}l l X@{}}
\toprule
\textbf{Symbol} & \textbf{Name} & \textbf{Description} \\
\midrule
$P \in \mathcal{P}$ & Principal & The human or organizational entity that delegated authority. $P \neq G$ is required. \\
$G \in \mathcal{G}$ & Agent & The computational entity performing the action, acting on behalf of $P$. \\
$\alpha \in \mathcal{A}$ & Action & The tool call being evaluated (function name + arguments). \\
$R \in \mathcal{R}$ & Resource & The target of the action (database, API endpoint, file). \\
$C$ & Context & Environmental attributes at evaluation time (timestamp, session ID, IP, metadata). \\
$H = [(a_1, d_1), \ldots, (a_k, d_k)]$ & History & Ordered sequence of prior (action, decision) pairs in the current session, bounded to window size $k$. \\
$\Pi = \{\pi_1, \ldots, \pi_n\}$ & Policy set & A finite set of rules, each mapping inputs to a decision. \\
\bottomrule
\end{tabularx}
\end{table}

This is paired with a capability judgment:

\begin{equation}
\Can(W, \alpha, s) \;\to\; \{\textsc{Feasible}, \textsc{Infeasible}\}
\label{eq:can}
\end{equation}

\noindent An action executes if and only if \emph{both} judgments are satisfied:

\begin{equation}
\mathit{Execute}(\alpha) \;\iff\; \Can(W, \alpha, s) = \textsc{Feasible} \;\wedge\; \May(P, G, \alpha, R, C, H, \Pi) = \Allow
\label{eq:execute}
\end{equation}

\subsection{Decision Lattice}

The decision domain $\mathcal{D}$ forms a chain lattice under the ordering $\Allow \sqsubseteq \Escalate \sqsubseteq \Deny$:

\begin{center}
\begin{tikzpicture}[node distance=0.8cm]
  \node (deny) {\Deny};
  \node[below=of deny] (esc) {\Escalate};
  \node[below=of esc] (allow) {\Allow};
  \draw[-] (allow) -- (esc);
  \draw[-] (esc) -- (deny);
  \node[right=0.6cm of deny, font=\small\color{midgray}] {(most restrictive)};
  \node[right=0.6cm of esc, font=\small\color{midgray}] {(defer to human)};
  \node[right=0.6cm of allow, font=\small\color{midgray}] {(least restrictive)};
\end{tikzpicture}
\end{center}

\noindent The join $d_1 \sqcup d_2$ returns the more restrictive decision: $\Allow \sqcup \Escalate = \Escalate$, $\Escalate \sqcup \Deny = \Deny$, etc. The meet $d_1 \sqcap d_2$ returns the less restrictive. This three-valued structure is the first extension beyond standard access control: most authorization systems are two-valued (permit/deny). XACML's four-valued output (\textsc{Permit}, \textsc{Deny}, \textsc{NotApplicable}, \textsc{Indeterminate}) conflates ``no policy applies'' with ``requires human judgment''~\citep{xacml2013}. $\Escalate$ is semantically distinct: it means the normative status is undetermined and requires human authority to resolve.

\subsection{Policy Composition}

We define the \emph{deny-overrides} composition operator $\oplus$:

\begin{definition}[Policy Composition]
\label{def:composition}
For policies $\pi_1, \pi_2$ and request $r$:
\begin{align}
(\pi_1 \oplus \pi_2)(r) &= \pi_1(r) \sqcup \pi_2(r)
\end{align}
For a policy set $\Pi = \{\pi_1, \ldots, \pi_n\}$:
\begin{align}
\oplus(\emptyset)(r) &= \Allow \quad\text{(identity element)} \\
\oplus(\Pi)(r) &= \bigsqcup_{\pi \in \Pi} \pi(r)
\end{align}
\end{definition}

\begin{proposition}
$(\mathcal{D}, \oplus)$ is a commutative monoid with identity $\Allow$.
\end{proposition}

\begin{proof}
Associativity: $\sqcup$ on a chain lattice is associative. Commutativity: $\sqcup$ on a chain is commutative. Identity: $\Allow \sqcup d = d$ for all $d \in \mathcal{D}$.
\end{proof}

\noindent This is an instance of the deny-overrides combining algorithm from XACML~\citep{xacml2013}, restricted to our three-valued domain. The restriction is deliberate: we trade XACML's generality (multiple combining algorithms) for analyzability (a single, well-understood operator with provable properties).

\subsection{Connection to Input/Output Logic}

Authority models fit naturally into the Input/Output (I/O) logic framework of Makinson and van der Torre~\citep{makinson2000input, makinson2003input}. In I/O logic, norms are not truth-valued propositions but ordered pairs $(a, x)$ where $a$ is an input (the factual antecedent) and $x$ is the output (the normative consequent). Given a norm set $G$ and input $A$, the output $\mathrm{out}(G, A)$ is derived by collecting the consequents of triggered norms and closing under logical consequence. This avoids the paradoxes of Standard Deontic Logic---Ross's paradox (obligation closed under disjunction), Chisholm's contrary-to-duty paradox (conflicting obligations from violated norms), and the Good Samaritan paradox (obligating the presupposition of an obligation)---which arise from treating norms as propositions closed under classical logic~\citep{vonwright1951deontic, sep-deontic}. In I/O logic, norms sit \emph{between} input and output as a transformation device; logical closure applies to inputs and outputs separately, not to the norms themselves.

An authority policy $\pi$ is an I/O pair: the input is the request tuple $(P, G, \alpha, R, C, H)$ and the output is $d \in \mathcal{D}$. The policy set $\Pi$ is a set of such pairs. Evaluating $\May$ is computing the output of a normative system given an input context---precisely the operation I/O logic formalizes. When policies conflict (one norm outputs \Allow, another \Deny{} for the same request), I/O logic's \emph{constrained output} filters through maximal consistent subsets---a conservative mechanism that, in our framework, maps naturally to \Escalate{} when genuine ambiguity remains.

The three-valued output maps to deontic operators:
\begin{itemize}[leftmargin=1.5em, itemsep=1pt]
  \item $\Allow \approx \mathsf{PE}(\alpha)$: the action is \emph{permitted}.
  \item $\Deny \approx \mathsf{IM}(\alpha) \equiv \mathsf{OB}(\neg\alpha)$: the action is \emph{impermissible}.
  \item $\Escalate$: the normative status of $\alpha$ is undetermined---a \emph{normative gap} in the sense of Alchourrón and Bulygin~\citep{alchourrón1971normative}, combined with a procedural default (defer to human authority). This has no direct counterpart in Standard Deontic Logic, where every action is either permitted or forbidden, but is well-studied in the normative systems literature.
\end{itemize}

\noindent Where policies conflict, we apply defeasible reasoning: the deny-overrides operator $\oplus$ is a specificity-free conflict resolution mechanism~\citep{governatori2004defeasible}. More complex priority schemes (temporal ordering, hierarchical specificity) can be layered on top.

\section{Formal Properties}
\label{sec:properties}

We prove three properties of authority models and establish decidability for a bounded policy language.

\subsection{Can/May Separation}

The central theorem states that capability and authority are independent: knowing what an agent \emph{can} do tells you nothing about what it \emph{may} do.

\begin{theorem}[Can/May Independence]
\label{thm:canmay}
Let $\Can: \mathcal{A} \to \{0, 1\}$ be a capability predicate and $\May: \mathcal{A} \times \Sigma \to \mathcal{D}$ be an authority function over policy-state space $\Sigma$. The systems are informationally independent: for any $c \in \{0,1\}$ and $d \in \mathcal{D}$, there exists a configuration $(\Can, \May, \sigma, \alpha)$ such that $\Can(\alpha) = c$ and $\May(\alpha, \sigma) = d$.
\end{theorem}

\begin{proof}
We exhibit configurations for all six cells of the $\{0,1\} \times \mathcal{D}$ product:

\begin{enumerate}[label=(\roman*), leftmargin=2em, itemsep=1pt]
  \item $\Can(\alpha) = 1 \wedge \May(\alpha, \sigma) = \Deny$. An agent possesses valid API credentials for a database ($\Can = 1$). Policy prohibits production modifications outside maintenance windows ($\May = \Deny$).
  \item $\Can(\alpha) = 0 \wedge \May(\alpha, \sigma) = \Allow$. Policy permits deployment to staging ($\May = \Allow$). The agent lacks SSH credentials ($\Can = 0$).
  \item $\Can(\alpha) = 1 \wedge \May(\alpha, \sigma) = \Allow$. The agent has credentials and the policy permits the action.
  \item $\Can(\alpha) = 0 \wedge \May(\alpha, \sigma) = \Deny$. The agent lacks credentials and the policy forbids the action.
  \item $\Can(\alpha) = 1 \wedge \May(\alpha, \sigma) = \Escalate$. The agent has tool access but the action's cost exceeds the automated approval threshold.
  \item $\Can(\alpha) = 0 \wedge \May(\alpha, \sigma) = \Escalate$. The agent lacks credentials and the policy requires human review for this action category.
\end{enumerate}

\noindent Since $\Can$ depends on tool availability (credentials, network access, installed software) and $\May$ depends on policy state (delegation scope, budget limits, temporal constraints), the two functions are defined over independent information sources. All six combinations are realizable.
\end{proof}

\begin{corollary}[Capability-Authority Conflation Attack]
\label{cor:conflation}
Any system where $\May(\alpha) = \Allow$ whenever $\Can(\alpha) = 1$---i.e., capability implies authority---admits an attack: acquiring a capability automatically grants authority, violating the principle of least authority~\citep{miller2006robust, dennis1966programming}. Formally, if $\forall \alpha: \Can(\alpha) = 1 \implies \May(\alpha) = \Allow$, then any capability escalation is simultaneously an authority escalation.
\end{corollary}

\begin{proof}
By contraposition. If capability implied authority, then an adversary who obtains tool access (e.g., via prompt injection that reveals an API key) would automatically be authorized for all operations the tool supports. This contradicts Theorem~\ref{thm:canmay}, which shows that capability and authority are independently configurable. The conflation eliminates the authority check entirely.
\end{proof}

\subsection{Deny-Monotonicity}

\begin{theorem}[Deny-Monotonicity]
\label{thm:deny-mono}
For any policy set $S \subseteq \Pi$ and policy $p \in \Pi$: if $\oplus(S)(r) = \Deny$, then $\oplus(S \cup \{p\})(r) = \Deny$.
\end{theorem}

\begin{proof}
$\oplus(S \cup \{p\})(r) = \oplus(S)(r) \sqcup p(r) = \Deny \sqcup p(r)$. Since $\Deny$ is the top element of $(\mathcal{D}, \sqsubseteq)$, we have $\Deny \sqcup d = \Deny$ for all $d \in \mathcal{D}$. Therefore $\oplus(S \cup \{p\})(r) = \Deny$.
\end{proof}

\begin{corollary}[Safety Preservation]
\label{cor:safety}
No additional policy can override an existing denial. Security-critical deny rules cannot be circumvented by adding permissive policies.
\end{corollary}

\subsection{Escalation Monotonicity}

\begin{theorem}[Escalation Monotonicity]
\label{thm:escalation}
For any policy set $S \subseteq \Pi$ and policy $p \in \Pi$: if $\oplus(S)(r) = \Escalate$, then $\oplus(S \cup \{p\})(r) \in \{\Escalate, \Deny\}$.
\end{theorem}

\begin{proof}
$\oplus(S \cup \{p\})(r) = \Escalate \sqcup p(r)$.

\noindent Case $p(r) = \Allow$: $\Escalate \sqcup \Allow = \Escalate$. $\checkmark$

\noindent Case $p(r) = \Escalate$: $\Escalate \sqcup \Escalate = \Escalate$. $\checkmark$

\noindent Case $p(r) = \Deny$: $\Escalate \sqcup \Deny = \Deny$. $\checkmark$

\noindent In all cases, $\oplus(S \cup \{p\})(r) \in \{\Escalate, \Deny\}$.
\end{proof}

\begin{corollary}[Human-in-the-Loop Preservation]
\label{cor:hitl}
No added policy can silently convert an escalation into an automatic approval. If any policy requires human review, only a more restrictive policy (denial) can override it.
\end{corollary}

\subsection{Decidability of \texorpdfstring{$L_{\mathrm{auth}}$}{L\_auth}}

\begin{definition}[$L_{\mathrm{auth}}$]
\label{def:lauth}
$L_{\mathrm{auth}}$ is a policy language where: (i) policies are finite sets of rules, (ii) each rule maps a Boolean condition over attributes from a finite domain to a decision $d \in \mathcal{D}$, (iii) history $H$ is bounded to a sliding window of size $k$, and (iv) conditions are Boolean combinations of attribute comparisons (equality, inequality, set membership, numeric bounds).
\end{definition}

\begin{theorem}[Decidability]
\label{thm:decidability}
For any well-formed policy $\Pi$ in $L_{\mathrm{auth}}$ and request $r$ with bounded history $|H| \leq k$, $\May(r, \Pi)$ is computable in time polynomial in $|\Pi|$ and $k$.
\end{theorem}

\begin{proof}[Proof sketch]
Each rule in $\Pi$ is a function from a finite attribute space to $\mathcal{D}$. Evaluating a single rule requires checking a Boolean condition over attributes---decidable in time linear in the condition size. The bounded history $H$ adds at most $k$ additional attribute lookups per rule. Composing $n = |\Pi|$ rules requires $n$ evaluations followed by $n-1$ join operations in $\mathcal{D}$. Total evaluation time: $O(n \cdot (|c| + k))$ where $|c|$ is the maximum condition size.

The restriction to bounded $H$ is necessary: Harrison, Ruzzo, and Ullman proved that safety of general access control systems is undecidable~\citep{harrison1976protection}. If $H$ were unbounded, policy evaluation could encode a Turing machine via the sequence of prior actions. The bound on $H$ limits the state space to $|\mathcal{D}|^k$, which is finite for fixed $k$.

$L_{\mathrm{auth}}$ is a proper subset of Cedar's decidable policy language~\citep{cutler2024cedar}, extended with bounded history as a finite relation. Cedar achieves decidability through similar restrictions: no recursion, finite attribute domains, and a strict type system.
\end{proof}

The architecture has the following properties:

\begin{itemize}[leftmargin=1.5em, itemsep=2pt]
  \item \textbf{Interception.} The SDK wraps an agent's tool set. Every proposed tool call is intercepted and evaluated against the policy store before execution. The agent is unaware of the authorization layer.
  \item \textbf{Deterministic evaluation.} Given a tool call and policy set, the decision ($\Allow$, $\Deny$, $\Escalate$) is reproducible. Policies are expressed in a declarative format that can be version-controlled and audited.
  \item \textbf{Escalation as legitimacy extension.} When a human approves an escalated action, they extend their delegated authority for that specific operation, creating an auditable chain from principal through agent to action.
  \item \textbf{Session-aware state.} The runtime tracks per-session call counts, argument history, and cross-tool constraints, enabling policies over action \emph{sequences}, not just individual calls.
  \item \textbf{Shadow mode.} A non-blocking mode evaluates and logs decisions without blocking tool calls, for gradual deployment and policy calibration.
\end{itemize}

  \item \textbf{P2 (Ambient):} Context makes violation natural (``the user seems to want this'').
  \item \textbf{P3 (Adversarial):} Prompt injection or social engineering attempting bypass.
\end{itemize}

\subsection{Metrics}

Two primary metrics capture the architectural distinction between deterministic and probabilistic enforcement:

\begin{definition}[Enforcement Determinism]
\label{def:ed}
$\mathrm{ED} = 1 - \mathrm{Var}(\text{decisions across } N \text{ identical runs})$. An $\mathrm{ED}$ of 1.0 means the same scenario always produces the same decision.
\end{definition}

\begin{definition}[Degradation Under Adversarial Pressure]
\label{def:dap}
$\mathrm{DAP} = \mathrm{VR}(P3) - \mathrm{VR}(P1)$, where $\mathrm{VR}$ is the violation rate. A $\mathrm{DAP}$ of 0.0 means adversarial pressure does not increase violations.
\end{definition}

\noindent Secondary metrics: Violation Rate (VR), Task Completion Rate (TCR), and False Denial Rate (FDR).

\subsection{Architectural Analysis}

Authority models achieve optimal values on both primary metrics \emph{by construction}:

\begin{proposition}
\label{prop:ed}
For any authority model with deterministic policy evaluation, $\mathrm{ED} = 1.0$.
\end{proposition}
\begin{proof}
Policy evaluation is a deterministic function: $\May(P, G, \alpha, R, C, H, \Pi)$ produces the same output for the same inputs. No LLM is in the evaluation path. Variance across identical runs is zero.
\end{proof}

\begin{proposition}
\label{prop:dap}
For any authority model with external enforcement, $\mathrm{DAP} = 0.0$.
\end{proposition}
\begin{proof}
Adversarial pressure (prompt injection, social engineering) affects the \emph{agent's} reasoning, not the authority engine's evaluation. Since enforcement is external to the model, adversarial manipulation of the agent does not change the policy evaluation. $\mathrm{VR}(P3) = \mathrm{VR}(P1) = 0$, so $\mathrm{DAP} = 0$.
\end{proof}

\noindent These results are architectural, not empirical. They follow from the design: removing the LLM from the authorization path eliminates stochastic variance and adversarial degradation. The interesting empirical questions concern the baselines: how badly do prompt-only and guardrail approaches degrade? Existing evidence suggests the degradation is severe: Agent-SafetyBench reports average safety scores below 60\%~\citep{zhang2024agentsafetybench}; OpenAgentSafety finds unsafe behavior in 51--73\% of safety-vulnerable tasks~\citep{vijayvargiya2025openagentsafety}; Potham reports an ``illusion of compliance'' where high adherence scores mask task incompetence~\citep{potham2025hierarchical}. We expect prompt-only baselines to show $\mathrm{ED} < 0.5$ and $\mathrm{DAP} > 0.4$ under P3 conditions.

\section{Related Work}
\label{sec:related}

Authority models build on five decades of access control, deontic logic, and institutional analysis. We trace the lineage honestly and identify what is genuinely new.

\subsection{Access Control Theory}

The access control matrix~\citep{lampson1971protection} established that authorization requires a reference monitor---a trusted component that mediates all access to objects. Anderson formalized the reference monitor concept: it must be tamper-proof, always invoked, and small enough to verify~\citep{anderson1972planning}. Our authority engine (\Cref{fig:architecture}) is a reference monitor for AI agent tool calls.

Harrison, Ruzzo, and Ullman proved that the safety question for general access control systems (``can a right ever leak to an unauthorized subject?'') is undecidable~\citep{harrison1976protection}. This result directly motivates our restriction to $L_{\mathrm{auth}}$ with bounded history (\Cref{thm:decidability}).

Role-Based Access Control (RBAC) assigns permissions to roles~\citep{sandhu1996rbac}. Constrained RBAC supports separation of duties and delegation, but roles are statically assigned---inadequate for agents whose authority depends on session context, conversation history, and dynamic delegation. Attribute-Based Access Control (ABAC) evaluates requests against policies over attributes~\citep{nist2014abac}, and is the closest structural ancestor to the $\May$ judgment. The $\May$ function is, structurally, an ABAC evaluation function with three specific extensions for the agentic context: (i)~the principal--agent separation ($P \neq G$) is first-class, not a derived attribute; (ii)~the $\Escalate$ output provides human-in-the-loop semantics absent from standard ABAC; and (iii)~the bounded history $H$ enables session-aware, stateful authorization.

Cedar provides a formally verified policy language with decidability guarantees~\citep{cutler2024cedar}. $L_{\mathrm{auth}}$ is a subset of Cedar's expressiveness, extended with session history. Zanzibar implements relationship-based access control at global scale~\citep{pang2019zanzibar} but lacks temporal constraints, budget semantics, and escalation. OPA/Rego offers a general-purpose policy engine but without formal decidability guarantees.

\noindent The contribution is not inventing authorization. It is identifying which extensions existing frameworks need for the agentic setting and proving that those extensions preserve desirable formal properties.

\subsection{Deontic Logic and Institutional Analysis}

Deontic logic, the logic of obligation, permission, and prohibition, originates with von Wright~\citep{vonwright1951deontic}. Standard Deontic Logic (SDL) treats norms as propositions, leading to well-known paradoxes. Input/Output logic~\citep{makinson2000input, makinson2003input} resolves these by treating norms as input/output pairs rather than truth-valued formulas. We adopt I/O logic as the conceptual substrate for authority models (\Cref{sec:authority}).

Sergot et al.\ demonstrated the feasibility of formalizing legal rules as executable programs in their encoding of the British Nationality Act~\citep{sergot1986british}---a direct ancestor of machine-checkable authority policies. Alchourrón and Bulygin formalized normative gaps~\citep{alchourrón1971normative}, which correspond to our $\Escalate$ output. Governatori and Rotolo developed defeasible deontic logic for conflict resolution~\citep{governatori2004defeasible}, which informs our deny-overrides composition.

Crawford and Ostrom's ADICO grammar~\citep{crawford1995grammar} decomposes institutional statements into Attributes, Deontic, aIm, Conditions, and Or-else. The $\May$ judgment is a computational ADICO statement (\Cref{tab:adico}). Ostrom's design principles for institutional governance~\citep{ostrom1990governing} provide normative criteria for evaluating authority model designs.

\subsection{Agent Safety and Verification}

Kambhampati et al.\ proved that LLMs cannot reliably self-verify plans~\citep{kambhampati2024llmmodulo} and that task success alone is insufficient for safe deployment~\citep{kambhampati2024tasksuccess}. We extend this to authorization: the authority model functions as a hard critic---in Kambhampati's terminology---for legitimacy, providing soundness guarantees the LLM cannot provide itself.

Agent-SafetyBench evaluates 16 LLM agents across 2,000 test cases with an average safety score of 38.5\%~\citep{zhang2024agentsafetybench}. OpenAgentSafety reports agents that ``disregard authorization entirely''~\citep{vijayvargiya2025openagentsafety}. ToolEmu finds even the safest agents exhibit severe failures 23.9\% of the time~\citep{ruan2024toolemu}. MCP-SafetyBench demonstrates that MCP's openness introduces new safety risks~\citep{zong2025mcpsafetybench}. These benchmarks measure \emph{safety} (harm avoidance), not \emph{authorization} (scope compliance). AuthorityBench (\Cref{sec:evaluation}) fills this gap.

\subsection{Emerging Agent Governance}

AgentSpec provides customizable runtime enforcement with a domain-specific policy language~\citep{wang2025agentspec}. ShieldAgent implements verifiable safety reasoning at execution time~\citep{chen2025shieldagent}. Both enforce pre-execution checks---confirming the action-boundary thesis---but neither formalizes the principal--agent--delegation--escalation semantics that authority models provide. Policy Cards propose machine-readable governance artifacts~\citep{mavracic2025policycards}. Agent Contracts formalize resource-bounded execution~\citep{ye2026agentcontracts}. South et al.\ develop authenticated delegation extending OAuth~2.0 and OpenID Connect~\citep{south2025authenticated, south2025identitymgmt, nagabhushanaradhya2025oidca}. Institutional AI reframes alignment as mechanism design~\citep{pierucci2026institutionalai, bracale2026institutionalcournot}.

Authority models sit between the identity layer (who is the agent?) and the policy specification layer (how is the policy expressed?), providing the \emph{decision function}---the runtime judgment that combines identity, delegation, policy, context, and history into $\{\Allow, \Deny, \Escalate\}$.

\section{Discussion and Limitations}
\label{sec:discussion}

\textbf{NL-to-policy reliability.} Natural language policy specification depends on an LLM translation step. This introduces a non-deterministic component upstream of the deterministic engine. Error rates, ambiguity handling, and contradictory policies require explicit design. The determinism guarantee (\Cref{prop:ed}) applies to \emph{evaluation}, not \emph{specification}.

\textbf{Policy completeness.} Authority models enforce stated policies. They cannot prevent violations of unstated policies. The guarantee is: if a policy exists, enforcement is deterministic. If no policy applies, the default decision (configurable; we recommend $\Deny$) takes effect.

\textbf{Principal-agent collusion.} The threat model assumes honest principals. If the principal and agent collude to circumvent policies, authority models cannot help---the principal can simply grant broader scope.

\textbf{Performance overhead.} Every tool call requires a policy evaluation round-trip. For the reference implementation, evaluation completes in single-digit milliseconds for typical policy sets. Latency may be significant for extremely high-frequency tool calls.

\textbf{Decidability boundary.} $\May$ with unbounded $H$ is undecidable (\Cref{thm:decidability}). Applications requiring unbounded history reasoning---cross-session correlation, long-horizon planning constraints---are outside $L_{\mathrm{auth}}$ and require approximation or human review.

\textbf{Bypass via non-tool-call actions.} Authority enforcement at the action boundary assumes all consequential agent actions pass through tool calls. Agents that generate executable code, compose tool calls into atomic plans, or communicate through channels outside the tool framework may bypass enforcement. This is a scope limitation, not a design flaw: as tool-use protocols mature, the action boundary becomes more comprehensive.

\section{Conclusion}
\label{sec:conclusion}

We defined authority models, proved three properties, and showed that deterministic policy enforcement achieves $\mathrm{ED} = 1.0$ and $\mathrm{DAP} = 0.0$ by construction. The Can/May separation is not a feature---it is an architectural requirement for governable AI agents.

Every major computing platform has developed a trust layer: operating systems developed file permissions and user isolation; the web developed TLS, OAuth, and CORS; cloud computing developed IAM, service accounts, and policy engines; mobile platforms developed app permissions and capability sandboxes. These trust layers were not new primitives invented from scratch---they were adaptations of access control concepts to new execution contexts. Authority models follow the same pattern: adapting and extending ABAC, deontic logic, and institutional analysis for the AI agent context, where actions are open-ended, tool-mediated, and selected by probabilistic reasoning.

The long-term challenge is not aligning isolated agents but building institutional structure for machine agency~\citep{chan2025infrastructure, pierucci2026institutionalai}. As agents move from generating text to exercising power, the decisive problem is whether their actions are properly constituted: attributable, scoped, constrained, reviewable, and revocable. Capability is not authority. Feasibility is not legitimacy. \Can{} is not \May{}.
